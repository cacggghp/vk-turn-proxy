# Рекомендации по улучшению клиента vk-turn-proxy

## 1. Управление ресурсами и эффективность
- [ ] **Оптимизация HTTP-клиента**: Вынести `http.Client` и `Transport` из замыкания `doRequest` в глобальную область или передавать как аргумент. Это позволит повторно использовать TCP-соединения (Keep-Alive) и избежать исчерпания сокетов.
- [ ] **Переиспользование буферов**: Использовать `sync.Pool` для аллокации пакетов `make([]byte, 1600)` в циклах чтения. Это снизит нагрузку на Garbage Collector при интенсивном трафике.

## 2. Конкурентность и синхронизация
- [ ] **Обработка Context**: Убедиться, что горутины чтения/записи немедленно завершаются при отмене контекста `ctx`, не дожидаясь срабатывания таймаутов `SetDeadline`.
- [ ] **Race Condition в addr**: Предусмотреть проверку на `nil` при первом вызове `addr.Load()` в цикле чтения из DTLS, чтобы избежать потенциальных ошибок, если сервер прислал пакет раньше первого пакета от клиента.

## 3. Надежность и обработка ошибок
- [ ] **Безопасный парсинг API**: Убрать `recover()` и паники из `getVkCreds`. Использовать проверку типов (`v, ok := resp["data"].(map[string]interface{})`) для всех полей ответов VK/OK API.
- [ ] **Конфигурация секретов**: Рассмотреть возможность выноса `client_secret` и `application_key` во внешние параметры или переменные окружения.

## 4. Безопасность
- [ ] **Проверка сертификатов**: Добавить возможность опциональной проверки сертификатов TURN-серверов (хотя для текущей схемы обфускации это может быть избыточно).

## 5. Мониторинг
- [ ] **Метрики**: Добавить счетчики переподключений и ошибок в логах для более удобной диагностики проблем в нестабильных сетях.
